# Provenance Guard

A backend service any creative-sharing platform can plug into to estimate whether
a piece of submitted text was **substantially generated by AI and passed off as a
person's original work** — then surface that honestly to readers and let creators
appeal. The hard problem here isn't detection accuracy (perfect AI detection is
unsolved); it's **behaving honestly when the system is unsure** and **not falsely
accusing real writers**.

Design decisions and the architecture diagram live in [`planning.md`](planning.md),
written before any implementation code.

## Architecture overview — the path a submission takes

```
POST /submit  {text, creator_id}
        │
        ├─► Signal 1: LLM classification (Groq llama-3.3-70b)   ──┐  semantic
        │        raw text ─► ai_likelihood 0–1                    │
        ├─► Signal 2: stylometry (pure Python)                  ──┤  structural
        │        raw text ─► ai_likelihood 0–1                    │
        │                                                         ▼
        │                                 scoring.combine() ─► confidence 0–1
        │                                  (weighted blend + disagreement pull)
        │                                                         │
        │                                  labels.classify_label()│
        │                                                         ▼
        │                          attribution band + transparency label text
        │                                                         │
        ├─────────────────────────► audit log (SQLite, append-only)
        ▼
   response {content_id, attribution, confidence, label}

POST /appeal {content_id, creator_reasoning}
        └─► append appeal row (status: under_review) ─► audit log ─► confirmation
```

A creator submits text to `/submit`. The raw text is sent to **both** signals
independently; each returns a 0–1 "how AI-like" score. `scoring.combine()`
normalizes them onto one calibrated `confidence`, then `labels.classify_label()`
maps that score to one of three attribution bands and its reader-facing label. The
full decision — both signal scores, the combined confidence, the attribution, a
text snippet, IDs and timestamp — is written to an append-only SQLite audit log,
and the response returns to the platform. Separately, `/appeal` takes a
`content_id`, appends an appeal record (flipping the content's current status to
`under_review`) **next to** the original decision, and confirms receipt.

## Detection signals

The system combines a **semantic** signal (the LLM) and a **structural** signal
(stylometry). They fail in different situations, so the combination is more
informative than either alone, and a confident verdict requires them to *agree*.
Stylometry itself measures three distinct properties; the *Ensemble detection*
stretch feature (below) promotes those three to independently-weighted members, so
the live system is a **4-member weighted ensemble** (LLM + 3 stylometric properties).

### Signal 1 — LLM classification (Groq), the *semantic* signal

- **What it measures:** holistic semantic and stylistic coherence — does the
  argument develop like a person's, does the register read "AI-polished?" It judges
  the text the way an attentive reader would.
- **Output:** `{verdict, ai_likelihood (0–1), reasoning}`, parsed from a low-temperature
  JSON-mode response.
- **Why chosen:** it captures meaning and overall feel, which pure statistics can't.
- **What it misses (blind spot):** it's an AI judging AI — uncalibrated (it will say
  0.8 when it's really unsure), non-deterministic, can be gamed by text written to
  "sound human," and tends to over-flag formal or non-native human writing. If the
  API errors or returns malformed JSON, the signal **degrades to a neutral 0.5**
  rather than guessing — an unavailable signal must never become a confident accusation.

### Signal 2 — stylometric heuristics (pure Python), the *structural* signal

Three metrics, each capturing a **distinct** property so they aren't redundant.
Each is reduced to a 0–1 "how AI-like" sub-score. These three are the structural
**ensemble members** (each independently weighted — see *Ensemble detection*); the
equal-weight mean of them is retained only as the aggregate `stylometric_score`
recorded in the audit log.

1. **Function-word profile (Burrows's Delta).** The per-text rate of ~40 common
   function words (the, of, and, to, …) is standardized (z-scored against a
   reference basis) and compared by Delta distance to a known-**human** profile and
   a known-**AI** profile built from bundled reference corpora
   ([`fixtures/`](fixtures/)). Sub-score = `d_human / (d_human + d_ai)`. *Why:*
   function words are the strongest classical authorship signal because writers use
   them unconsciously; AI uses them at more uniform rates than an individual human's
   idiosyncratic pattern.
2. **Punctuation density** — punctuation marks per sentence. *Why:* humans are
   idiosyncratic about punctuation; AI tends to be tidy and predictable.
3. **Sentence-length variance (burstiness)** — std of sentence lengths. *Why:*
   "burstiness" is a named pillar of AI-text detection — humans write in bursts of
   long and short sentences; machine text trends uniform.

   *(Metrics 2 and 3 are mapped to 0–1 by linear interpolation between the
   human-mean and AI-mean **anchors** computed from the reference corpora, so the
   direction of each metric is learned from data, not hardcoded. Type-token ratio
   was considered and dropped — it saturated near 0.88 on short inputs and
   separated nothing.)*

- **What it misses (blind spot):** polished/formal human writers look "too clean"
  (false-positive risk); lightly-edited AI drifts toward "human"; it's unreliable on
  very short text (falls back to 0.5 below ~15 words); and it is blind to meaning.
  This is exactly why it's paired with the semantic signal.

## Confidence scoring

One score, `confidence` ∈ [0, 1] (0 = clearly human, 1 = clearly AI). Every ensemble
member already reports on this scale, so the combination is a **weighted average of
the four members plus a conflict pull** (see *Ensemble detection* for the per-member
weights and the math):

- The LLM is the most informative member (weight 0.55) but uncalibrated; the
  structural members are deterministic but weaker on short text.
- **Conflict pull:** the weighted mean is pulled toward 0.5 (uncertain) in
  proportion to how much the members' evidence **cancels** (`1 − net/gross`). When
  the members all lean the same way, nothing is pulled; when they split, the result
  is pulled into the uncertain band. A member near 0.5 carries little evidence, so
  it neither manufactures nor masks conflict. The upshot: **a confident verdict
  requires the members to agree** — the system's core honesty property.

What specific scores mean to a reader (`provenance/config.py`):

| `confidence` | attribution | what the reader is told |
|---|---|---|
| `0.00 – 0.40` | `likely_human` | little sign of AI |
| `0.40 – 0.70` | `uncertain` | couldn't tell — no conclusion drawn |
| `0.70 – 1.00` | `likely_ai` | strong signs of AI; appeal path offered |

The thresholds are **intentionally asymmetric**: the bar to say "likely AI" (0.70)
sits further from the middle than the "likely human" bar (0.40), giving human
writers more room before any accusation, because a false positive (accusing a real
person) is the costliest error on a creative platform.

### Validating that the scores are meaningful

I ran the pipeline against the spec's labelled test texts with the **live** Groq
signal and checked that scores spread the way intuition says they should — and that
all three bands are reachable. Two example submissions with noticeably different
confidence (lifted directly from that run):

| Example | `confidence` | attribution | ensemble members (llm / function_words / punctuation / burstiness) |
|---|---|---|---|
| **High** — AI-style listicle ("Maintaining a healthy work-life balance…") | **0.805** | `likely_ai` | 0.80 / 0.57 / 1.00 / 1.00 |
| **Low** — casual human review ("ok so i finally tried that ramen place…") | **0.144** | `likely_human` | 0.10 / 0.45 / 0.00 / 0.00 |

A third case is the most instructive. The spec's "clearly AI" essay ("Artificial
intelligence represents a transformative paradigm shift…") scores **0.617 →
`uncertain`**: the LLM is confident it's AI (0.80) but the structural members can't
corroborate on a single short paragraph (function_words 0.55, punctuation 0.36,
burstiness 0.38), so the members split and the system **honestly hedges instead of
accusing**. When a *longer* AI text gives the structural members enough to work with
(the listicle above — punctuation and burstiness both hit 1.00), they agree with the
LLM and the score rises to 0.805 `likely_ai`. That contrast — same "obviously AI to
a human" intuition, different honest outputs depending on how much evidence the
structural members actually have — is the system working as designed, not a bug.

The calibration ordering is also pinned as a **regression test**
([`tests/regression/test_calibration.py`](tests/regression/test_calibration.py)) so
a future change can't silently re-order clearly-human vs clearly-AI text.

## Ensemble detection (stretch feature)

The detector is a **4-member weighted ensemble** — three or more signals combined
by a documented weighting scheme. The members are the LLM plus the three stylometric
properties, each promoted to a first-class, independently-weighted member (they
measure genuinely distinct properties, so this is a real ensemble, not relabeling):

| Member | Property measured | Weight | Why this weight |
|---|---|---|---|
| `llm` | holistic semantic/stylistic coherence | **0.55** | the single most informative signal |
| `function_words` | unconscious word-choice habits (Burrows's Delta) | **0.20** | strongest *classical* stylometric signal |
| `burstiness` | sentence-length variance | **0.15** | a named pillar of AI-text detection |
| `punctuation` | punctuation density | **0.10** | a softer structural cue |

Weights live in `config.ENSEMBLE_WEIGHTS` and sum to 1.0.

**The weighting/voting rule** (`scoring.combine_ensemble`): the weighted mean of the
members, pulled toward 0.5 in proportion to how much their evidence cancels —

```
net      = | Σ wᵢ·(sᵢ − 0.5) |     # how strongly the members agree on a direction
gross    =   Σ wᵢ·|sᵢ − 0.5|        # total evidence regardless of direction
conflict = 1 − net/gross            # 0 = all lean the same way, 1 = evidence cancels
confidence = mean + (0.5 − mean) · conflict · 0.6
```

This generalizes the original 2-signal disagreement pull to N members. When all
members lean the same way `net == gross`, conflict is 0, and a confident verdict
stands; when they split, the evidence cancels and the result is pulled to uncertain.
A member near 0.5 contributes little `|sᵢ − 0.5|`, so it can't manufacture or mask
conflict. (The 2-signal `combine()` is preserved as the special case of this rule.)

**Visible in the output.** Every `/submit` response includes the per-member
breakdown, so all four votes are inspectable:

```json
// POST /submit  (the AI-style listicle)
{
  "content_id": "50f9474c-…",
  "attribution": "likely_ai",
  "confidence": 0.8048,
  "label": "Likely AI-generated. …",
  "signals": { "llm": 0.8, "function_words": 0.574, "punctuation": 1.0, "burstiness": 1.0 }
}
```

How the ensemble changed the calibration vs the base 2-signal blend (same texts,
live Groq): the high/low cases stayed firmly in their bands (0.823→0.805,
0.119→0.144) and the short AI essay still honestly hedges (0.639→0.617 uncertain) —
because three structural members splitting against a confident LLM expresses the
"not enough corroboration" state more faithfully than one bundled stylometry score
did. Structural invariants (≥3 members, weights sum to 1) are pinned by
[`tests/regression/test_ensemble.py`](tests/regression/test_ensemble.py).

## Transparency label

The label is what a reader sees on the platform. It communicates the result in
plain language, never makes a hard authorship claim, always admits fallibility, and
(for the AI variant) names the appeal path. The exact text of all three variants
(pinned verbatim in [`provenance/labels.py`](provenance/labels.py) and by a
regression test):

> **High-confidence human** (`confidence < 0.40`):
> "Likely human-written. Our automated check found little sign of AI generation in
> this piece. Automated checks aren't perfect — this is a signal, not a verdict."

> **Uncertain** (`0.40 ≤ confidence < 0.70`):
> "Uncertain. Our check couldn't confidently tell whether a person wrote this or AI
> generated it. This is common for short pieces, or for writing a person wrote and
> then used AI to edit. No conclusion is being drawn."

> **High-confidence AI** (`confidence ≥ 0.70`):
> "Likely AI-generated. Our automated check found strong signs this text was
> produced or heavily shaped by AI. This is an automated assessment, not a
> certainty — if you wrote this yourself, you can appeal."

## Appeals workflow

**Who can appeal:** the creator of a flagged submission, when they believe a
`likely_ai` or `uncertain` result is wrong.
**What they provide:** the `content_id` and `creator_reasoning` (free text).

`POST /appeal` then (1) appends an appeal record that flips the content's current
status to `under_review`, (2) logs it in the audit log **alongside** the original
decision — the original classification row is preserved (append-only), with the
original's scores copied onto the appeal row so a reviewer sees everything in one
record, and (3) returns a confirmation. Unknown `content_id` → `404`; missing
fields → `400`. Automated re-classification is intentionally **not** performed —
appeals route to human review.

Live example:

```json
// POST /appeal {"content_id": "dc40e339-…", "creator_reasoning": "I wrote this myself…"}
{
  "content_id": "dc40e339-9c51-49b6-bb1e-8610a6ef4789",
  "message": "Appeal received. This content has been logged for human review.",
  "status": "under_review"
}
```

## Rate limiting

Applied to `POST /submit` via Flask-Limiter, keyed per client IP, in-memory storage.

**Chosen limits: `10 per minute; 100 per day` per client.**

Reasoning (not arbitrary):
- **`/submit` is an expensive write** — every call hits the Groq LLM — so it belongs
  in the strict tier (industry norm ~10–30/min for expensive writes vs 60–300/min
  for cheap reads).
- **10/min = burst protection.** A creator submitting their own original writing
  posts a handful of pieces, never dozens a minute, so 10/min is generous for any
  real human while a flooding script hits the wall within seconds.
- **100/day = sustained-abuse cap + cost control.** 100 original submissions in a day
  is already an enormous amount of genuine writing; this stops a slow-drip adversary
  staying under the per-minute radar and bounds Groq API spend.
- **What I'd improve:** key on `creator_id` (or an auth token) in production rather
  than IP, since IPs can be shared (offices, NAT) or rotated.

Past the limit the endpoint returns a structured `429`:

```
# 12 rapid submits (limit 10/min); the first calls succeed, the rest are blocked:
200 200 200 200 200 200 200 429 429 429 429 429
```
```json
{
  "error": "rate_limit_exceeded",
  "message": "Submission rate limit exceeded (10 per 1 minute). Please slow down."
}
```
*(Seven 200s above rather than ten because three earlier submissions from the same
IP already counted against the per-IP, per-minute window — exactly the intended
cumulative behavior.)*

## Groq API usage & limits

Only **Signal 1** calls Groq — exactly one request per `/submit`. Signal 2
(stylometry) is pure Python, and `/appeal` and `/log` never call Groq.

Groq's free tier limits both **requests** and **tokens**, per minute and per day.
For `llama-3.3-70b-versatile`, the binding limits are:

| Limit | Value | Notes |
|---|---|---|
| Requests / minute | ~30 | Our own limiter (10/min) is stricter, so this is rarely the constraint |
| Tokens / minute | 12,000 | **The constraint that matters for long documents** |

> **⚠️ These limits are as of 2026-06-29 and may change.** Groq adjusts free-tier
> quotas over time — check your current values in the [Groq console](https://console.groq.com)
> (Dashboard → Metrics, "Show Limits") rather than relying on the figures above.

**Why the token limit matters, and the safeguard.** Because the whole submission
is sent to the model as input tokens, a long document costs far more than a short
poem. At our max request rate (10/min), staying under 12,000 tokens/min leaves a
budget of ~1,200 tokens (~900 words) per submission. To prevent a long document
from exceeding the per-minute token limit, the LLM signal **truncates its input to
the first `LLM_MAX_INPUT_WORDS` words** (default **1,200**, set in
[`provenance/config.py`](provenance/config.py)) — **stylometry still analyzes the
full text**, so only the semantic signal sees the capped version. If a Groq limit
is hit anyway, `llm_signal` degrades gracefully to a neutral `0.5` (the submission
still succeeds; that verdict then leans on stylometry alone).

## Known limitations

1. **Formal / academic human writing (a false positive — the costly direction).**
   A careful essayist writes with controlled sentence lengths, precise vocabulary,
   and consistent punctuation. Signal 2's core assumption — "uniform structure =
   AI" — misfires here, and Signal 1 also tends to over-flag formal register. In the
   live calibration, the formal-human monetary-policy passage scored higher on
   stylometry (aggregate 0.30) than the casual human review (0.15) — drifting toward
   AI precisely for the kind of writer the platform most wants to protect. The wide
   uncertain band and the asymmetric 0.70 AI threshold are the mitigations, but this
   is the system's most important failure mode.
2. **AI text lightly edited by a human (a false negative).** Varying a few sentence
   lengths and swapping some words moves the stylometrics toward "human" without
   changing that the content was substantially AI-generated — because Signal 2 only
   measures surface structure, which is exactly what light editing alters. Signal 1
   can be talked past the same way.
3. **Very short text** (poems, single lines): below ~15 words stylometry can't
   measure reliably and falls back to 0.5, so the verdict leans entirely on the LLM.

## Spec reflection

- **One way the spec helped.** Locking the label wording and the asymmetric
  `0.40 / 0.70` thresholds in `planning.md` *before* coding made the label and
  scoring modules unambiguous to implement — I lifted the three label strings
  straight from the spec into a regression test that pins them verbatim, so wording
  can't drift. Deciding "what 0.5 should mean to a user" up front (a wide,
  no-conclusion band) drove the threshold math rather than the other way around.
- **One way the implementation diverged.** `planning.md` sketched the disagreement
  pull as `blended + (0.5 - blended) · |s1 − s2| · k`, using the raw gap between the
  signals. During M4 live calibration that over-penalized a *non-committal*
  stylometry score: the clearly-AI essay (LLM 0.80, stylometry 0.43) was dragged
  down to 0.62, because a stylometry score of 0.43 — really "I can't tell" — was
  being treated as a strong disagreement. I diverged to a **cross-0.5 conflict**
  formulation: the pull fires only when the signals are on opposite sides of 0.5 and
  is scaled by the *weaker* signal's distance from 0.5. After the change, agreeing
  signals reach `likely_ai` (the listicle, 0.805) while genuine conflict still lands
  `uncertain`. (The *Ensemble detection* stretch feature later generalized this same
  conflict pull from two signals to four via the `net/gross` evidence ratio.)

## AI usage

1. **Choosing the stylometric signal.** I directed the AI to explain the candidate
   metrics (sentence-length variance, type-token ratio, punctuation density) and
   ground them in real prior art. It produced a live demo showing type-token ratio
   saturated (~0.88) on short inputs and separated nothing, plus references to
   classical stylometry (Mosteller & Wallace 1963; function words as the strongest
   authorship signal) and modern AI detection (burstiness, DetectGPT). **I overrode
   the spec's default pairing** (length variance + TTR): I dropped TTR for
   **burstiness** and added **function-word frequency via Burrows's Delta**, and
   reframed the function-word rationale specifically for human-vs-AI ("AI uses
   function words at more uniform rates than an individual human's pattern").
2. **The scoring blend.** I asked the AI to implement the confidence blend per the
   spec's disagreement-pull sketch. It produced a working version, but **live
   calibration that I ran revealed** it pushed a clearly-AI text into "uncertain"
   because it treated a near-neutral stylometry score as conflict. **I diagnosed the
   cause and revised** the pull to fire only on genuine cross-0.5 disagreement scaled
   by the weaker signal — then re-validated against live Groq that all three bands
   were reachable. (Full record in `planning.md` → "AI usage log".)
